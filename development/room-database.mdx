## Overview

Casa de Historias uses Room, Android's SQLite abstraction library, to provide offline access to stories. Room enables users to:
- Browse stories without an internet connection
- Cache downloaded stories locally
- Mark stories as favorites offline

## Architecture Components

The Room implementation consists of three main components:

1. **Database:** `AppDatabase` - The main database holder
2. **Entity:** `StoryEntity` - The data model for local storage
3. **DAO:** `StoryDao` - Data Access Object for database operations

```kotlin
    val contentNahuatl: String,
    val audioPath: String?, // Ruta local al archivo de audio

    val narratorName: String,
    val community: String,
    val isFavorite: Boolean = false
)
```

### Schema Comparison

<Note>
Notice the differences between `StoryEntity` (local) and `StoryFirestore` (cloud):

**Local Storage:**
- Uses `audioPath` and `imagePath` for local file paths
- Includes `isFavorite` flag for offline favoriting
- Excludes cloud-specific fields like timestamps and author info

**Cloud Storage:**
- Uses `audioUrl` and `imageUrl` for Firebase Storage URLs
- Includes metadata like `createdAt`, `updatedAt`, `authorId`
- Stores publication status and tags
</Note>

## Common Patterns

### Caching Strategy

Implement a repository pattern to coordinate between Firestore and Room:

```kotlin
@Singleton
class StoryRepository @Inject constructor(
    private val firestoreRepository: FirestoreRepository,
    private val storyDao: StoryDao,
    private val networkMonitor: NetworkMonitor
) {
    
    fun getStories(): Flow<List<Story>> = flow {
        // First, emit cached data for fast display
        val cachedStories = storyDao.getAllStories().first()
        if (cachedStories.isNotEmpty()) {
            emit(cachedStories.map { it.toDomain() })
        }
        
        // Then fetch fresh data if online
        if (networkMonitor.isOnline()) {
            try {
                val freshStories = firestoreRepository.getAllStories()
                
                // Update cache
                storyDao.insertStories(freshStories.map { it.toEntity() })
                
                // Emit fresh data
                emit(freshStories.map { it.toDomain() })
            } catch (e: Exception) {
                // Network error - stick with cache
            }
        }
    }
}
```

### Favorites Management

```kotlin
suspend fun toggleFavorite(storyId: String) {
    val story = storyDao.getStoryById(storyId)
    story?.let {
        val updated = it.copy(isFavorite = !it.isFavorite)
        storyDao.insertStories(listOf(updated))
        
        // Sync to Firestore if online
        if (networkMonitor.isOnline()) {
            if (updated.isFavorite) {
                firestoreRepository.addToFavorites(userId, storyId)
            } else {
                firestoreRepository.removeFromFavorites(userId, storyId)
            }
        }
    }
}
```

## Database Migrations

When you need to change the schema, create migrations:

```kotlin
val MIGRATION_1_2 = object : Migration(1, 2) {
    override fun migrate(database: SupportSQLiteDatabase) {
        // Example: Add a new column
        database.execSQL(
            "ALTER TABLE stories ADD COLUMN downloadedAt INTEGER DEFAULT 0 NOT NULL"
        )
    }
}

// In DatabaseModule
@Provides
@Singleton
fun provideDatabase(@ApplicationContext context: Context): AppDatabase {
    return Room.databaseBuilder(
        context,
        AppDatabase::class.java,
        "casa_historias_db"
    )
    .addMigrations(MIGRATION_1_2)
    .build()
}
```

<Warning>
Always provide migrations in production apps. Using `fallbackToDestructiveMigration()` will delete all user data when the schema changes.
</Warning>

```kotlin
class StoryDaoTest {
```

## Best Practices

<Steps>
  <Step title="Use Flow for reactive queries">
    Return `Flow` from DAO queries to automatically update UI when data changes.
  </Step>
  
  <Step title="Keep entities simple">
    Store only essential data locally. Don't replicate the entire cloud schema.
  </Step>
  
  <Step title="Implement proper migrations">
    Never use `fallbackToDestructiveMigration()` in production unless acceptable to lose user data.
  </Step>
  
  <Step title="Cache strategically">
    Don't cache everything. Consider caching:
    - Recently viewed stories
    - User's favorite stories
    - Stories from user's community
  </Step>
  
  <Step title="Clean up old cache">
    Implement a cache expiration strategy:
    
    ```kotlin
    @Query("DELETE FROM stories WHERE downloadedAt < :timestamp AND isFavorite = 0")
    suspend fun deleteOldCache(timestamp: Long)
    ```
  </Step>
</Steps>

## Performance Considerations

<AccordionGroup>
  <Accordion title="Indexing">
    Add indexes for frequently queried columns:
    
    ```kotlin
    @Entity(
        tableName = "stories",
        indices = [
            Index(value = ["community"]),
            Index(value = ["isFavorite"])
        ]
    )
    data class StoryEntity(...)
    ```
  </Accordion>
  
  <Accordion title="Batch Operations">
    Use batch inserts instead of inserting one by one:
    
    ```kotlin
    // Good
    storyDao.insertStories(listOfStories)
    
    // Bad
    listOfStories.forEach { story ->
        storyDao.insertStory(story)
    }
    ```
  </Accordion>
  
  <Accordion title="Limit Result Size">
    For large datasets, use LIMIT in queries:
    
    ```kotlin
    @Query("SELECT * FROM stories ORDER BY downloadedAt DESC LIMIT :limit")
    fun getRecentStories(limit: Int = 50): Flow<List<StoryEntity>>
    ```
  </Accordion>
</AccordionGroup>
